# 信号系统全局可调试（DevTools Data 面板）计划

## 目标
- 让 `src/core/reactivity.ts` 的 Signal/Effect 在运行期可被全局 DevTools 读取（只读为主）。
- 新增一个 `states.ts` 作为“调试状态汇聚层”，统一提供 snapshot/树形视图数据。
- 重写 `src/ui/window/developer/panels/data_panel.ts`：从 `states.ts` 获取状态树，渲染为可浏览的树状列表（先做最小可用：展开/折叠 + 基本文本展示）。

## 设计原则
- **零侵入业务代码**：业务代码不需要手动“注册”每个 signal，默认 `signal()` 创建即进入调试注册表；允许可选的命名/分组增强。
- **调试开关可控**：默认开启（开发态），后续可通过编译常量或 runtime flag 禁用，以避免生产环境泄露内部状态。
- **只读优先**：Data 面板初期只展示；后续再扩展“编辑值/执行表达式”等能力。
- **不引入重依赖**：保持现有 TS + Canvas UI 风格，不引入第三方 state devtools。

## 现状梳理（关键约束）
- `signal<T>()` 当前内部维护 `subs: Set<EffectFn>`，并在对象上挂了 `_subs`（见 `reactivity.ts`），这已经能帮助我们进行依赖图的基本推断。
- `effect()` 的 runner 有 `deps?: Set<Signal<unknown>>`，同样可用于反向推断（signal → effects / effect → signals）。
- 当前缺少：
  - signal 的稳定 ID（跨刷新不保证，但运行期稳定即可）
  - 可读的名称/作用域（scope）
  - 全局入口（DevTools 能在不 import 内部模块的情况下访问）

## 方案概览

### 1) reactivity.ts：增加“调试注册表”与可选元信息
在 `src/core/reactivity.ts` 中新增：
- 全局递增 `signalId`，每个 `signal()` 创建时生成 `id`。
- 一个全局 registry（例如 `Map<number, SignalDebugRecord>`），记录：
  - `id`
  - `value`（通过 `peek()` 获取；注意不要存拷贝，存 getter）
  - `subs`（已存在 `_subs`）
  - 可选 meta：`name?`、`scope?`、`createdAt?`
- 提供最小公开 API（供 `states.ts` 调用）：
  - `listSignals(): SignalDebugRecord[]`（或 iterator）
  - `setSignalMeta(sig, { name, scope })`（可选；供业务或框架层标注）
  - `getSignalMeta(sig)`（可选）

### 2) 全局 DevTools 入口（globalThis）
在 `reactivity.ts` 中以非侵入方式暴露调试接口，例如：
- `globalThis.__TNL_DEVTOOLS__ = { reactivity: { listSignals, ... } }`

约束：
- 仅暴露**只读**快照方法（本阶段不提供 setValue）。
- 避免把真实 value 直接 stringify（可能很大或包含循环引用）；由 `states.ts` 负责安全序列化。

### 3) 新增 states.ts：构建“面向面板”的状态树
新增 `src/ui/window/developer/states.ts`（或更通用的位置如 `src/devtools/states.ts`，按现有目录风格选择）：
- 从 `globalThis.__TNL_DEVTOOLS__` 或直接 import `reactivity.ts` 的 list API 获取 signals。
- 生成 Data 面板使用的结构：
  - `DataNode = { id, label, kind, children?, valuePreview? }`
  - 分组策略：
    - 一级：scope（如 `core` / `ui` / `timeline` / `media` / `unknown`）
    - 二级：name 前缀或模块名（后续增强）
    - 叶子：signal 节点（展示 `id/name/valuePreview/subscribersCount`）
- 提供安全的 value preview：
  - 对 primitive：直接展示
  - 对对象/数组：展示 `Array(n)` / `{...}` 并限制深度与长度
  - 对函数：`[Function]`
  - 对含循环引用：`[Circular]`

API：
- `getStateTree(): DataNode[]`
- `getStateSnapshotById(id): { value, deps, subs }`（后续 Inspector 用）

### 4) 重写 data_panel.ts：渲染树形 UI
在 `src/ui/window/developer/panels/data_panel.ts`：
- 替换当前 `TextSurface` 占位为一个真正的 `Surface`：
  - 使用 `ViewportElement + Scrollbar`（若需要）或复用现有滚动模型
  - 先实现最小交互：
    - 点击节点展开/折叠
    - 展示 valuePreview
    - 简单搜索（可延后）
- 数据刷新策略：
  - 初期：每次 `render()` 读取 `getStateTree()`（可接受）
  - 下一步：引入轻量缓存 + 变更时刷新（与 Timeline 面板埋点结合）

## 实施步骤（执行顺序）
1. 修改 `reactivity.ts`：实现 signal id、registry、list API；并在开发态挂到 `globalThis.__TNL_DEVTOOLS__`。
2. 新增 `states.ts`：实现 state tree 构建与 valuePreview 安全序列化。
3. 重写 `data_panel.ts`：实现可展开树形列表渲染与最小交互。
4. 验证：
   - 打开 DeveloperToolsWindow → Data 面板不报错
   - signals 数量、scope 分组展示正确
   - 展开/折叠交互正常，性能可接受
   - 不在控制台泄露大对象/敏感信息（仅 preview）

## 风险与缓解
- **内存泄漏**：registry 会持有 signal 引用；需要允许后续实现 `disposeSignal`（暂不做，但预留）。
- **隐私/敏感信息**：valuePreview 必须限深度、限长度；后续加生产态禁用。
- **性能**：signals 很多时树构建会重；先做分页/虚拟列表（后续 TODO）。
